<task type="auto">
  <name>Task 3: Evaluate --mlock and update startup flags if beneficial</name>
  <files>docs/benchmarks.md, patches/install_llama_server.sh</files>
  <action>
Evaluate --mlock for llama-server and document findings in docs/benchmarks.md. Then update install_llama_server.sh accordingly.

**What --mlock does:**
`--mlock` (maps to `mlock()` syscall) pins model pages in physical RAM after first fault. On the SAME worker instance: second cold start skips re-paging since pages are already pinned. On a NEW worker (scale-to-zero boot): no benefit — pages were never loaded.

**VRAM context for this model:**
- Model weights: ~60 GB VRAM (mmap'd from volume)
- KV cache: ~4-8 GB VRAM (at 32768 ctx)
- CUDA overhead: ~3-5 GB
- Total: ~67-73 GB of 97 GB VRAM used
- Remaining: ~24-30 GB headroom

**mlock evaluation:**
- `--mlock` pins pages in system RAM, not VRAM. GPU tensors are already in VRAM.
- For GGUF mmap: CPU pages (the memory-mapped file) get pinned, but the bottleneck is DMA transfer to VRAM, not page faults.
- **Verdict:** --mlock is unlikely to help for VRAM load time. The bottleneck is volume read speed → DMA to GPU, not CPU page fault resolution.

However, `--mlock` DOES help if the model is on local NVMe (prevents eviction). On network volume with mmap, it may slow things down by forcing page population upfront rather than on-demand.

**Decision:**
- Do NOT add --mlock to auto_warmup() in install_llama_server.sh
- Document why in docs/benchmarks.md under "--mlock Evaluation" section
- Note: if RunPod ever offers local NVMe model storage, --mlock should be re-evaluated

Add "--mlock Evaluation" section to docs/benchmarks.md with this analysis.

**No change to install_llama_server.sh** — leave auto_warmup() flags as-is (mmap without mlock is correct for network volume).

Existing auto_warmup() flags to preserve:

**Also add a "Cost of Warmup" section to docs/benchmarks.md:**

RTX Pro 6000 Blackwell on RunPod: $1.69/hr

Warmup cost per cold start (assuming ~10 min to model ready):
```
$1.69 / 60 * 10 = $0.28 per warmup
```

Usage model — a human developer with AI assistance (no agentic loops):
- Realistic AI utilization: ~35% of coding time (writing prompts, reviewing output, thinking in between)
- 40h/week × 4.33 weeks = ~173h/month
- Active AI sessions: ~173 × 0.35 ≈ 60h/month
- Scale-to-zero means every session start triggers a warmup. Typical pattern: ~4 cold starts/workday (morning, post-lunch, after breaks/context switches)
- 22 workdays × 4 warmups = ~88 warmups/month

Monthly warmup cost: 88 × $0.28 ≈ **$24.60/month** (warmup overhead only, before inference GPU-seconds)

Comparison:
| Usage pattern | Warmups/month | Warmup cost | Notes |
|---|---|---|---|
| Light (1/day) | 22 | $6.16 | One session per workday |
| Typical (4/day) | 88 | $24.64 | 4 context switches/day |
| Heavy (8/day) | 176 | $49.28 | Pair programming / agentic loops |
| Always-on (idle_timeout=∞) | 1 | $0.28 | One boot, never scales to zero |

**Key insight:** If you're doing 4+ sessions/day, always-on (`idle_timeout` set high enough to span the workday) is cheaper than scale-to-zero. Break-even vs always-on for a workday:
- Always-on 9h: 9 × $1.69 = **$15.21/day**
- Scale-to-zero at 4 warmups: 4 × $0.28 + inference ≈ **$1.12/day** (warmup only) + whatever inference actually costs

So scale-to-zero wins unless you're keeping the endpoint hot continuously. Document recommended `idle_timeout` values:
- `idle_timeout=300` (5 min): aggressive scale-to-zero, cheapest for sporadic use
- `idle_timeout=3600` (1h): balances cost vs warmup for a focused coding session
- `idle_timeout=28800` (8h): stays warm for a full workday, ~$15/day if you use it or not

<task type="auto">
  <name>Task 4: Fix first-request failure with slot priming in /health</name>
  <files>nemotron.py</files>
  <action>
Modify the `gpu_health()` endpoint in nemotron.py so that when llama-server first transitions to status="ok", it sends a short priming completion request to slot 0 before returning status="ready" to the caller.

**The bug:** Nemotron-3-Super uses a hybrid attention + Mamba2 recurrent architecture (NemotronH). llama-server reports "ok" when tensors are loaded, but the first real completion request triggers:
  `forcing full prompt re-processing due to lack of cache data (see PR #13194)`
This causes the first request to fail with a garbled or empty response. The second request works fine.

**The fix:** After llama-server first reports "ok", send one priming completion request to 127.0.0.1:8081 before returning status="ready". This absorbs the KV cache initialization for slot 0. Subsequent real requests hit a warm slot.

**Implementation:**

Add a module-level flag `_slot_primed = False` to gpu_health(). Use `nonlocal` won't work across requests — store as an app state attribute: `gpu_api.state.slot_primed = False` (set in app startup or on first access).

In `gpu_health()`, after detecting `llama_ready = True`:
1. Check `getattr(gpu_api.state, "slot_primed", False)`
2. If not yet primed: send a short POST to `http://127.0.0.1:8081/v1/chat/completions` with:
   ```json
   {
     "model": "nemotron",
     "messages": [{"role": "user", "content": "hi"}],
     "max_tokens": 1,
     "temperature": 0
   }
   ```
   Use `httpx.AsyncClient` with `timeout=30.0` (slot init can take a few seconds).
3. Set `gpu_api.state.slot_primed = True` regardless of priming response (success or error — the point is to trigger the cache init, not to get a valid response).
4. Return `{"status": "ready"}` as before.

If priming fails with an exception, log it but still mark primed=True and return ready — better to let a real request fail once than to loop forever.

Add a comment referencing the llama.cpp issue: `# Prime slot 0 to absorb NemotronH hybrid-attention first-request KV cache init (llama.cpp PR #13194)`

Do NOT change the health check logic for the loading/cold states — only add priming logic in the `llama_ready=True` branch.
