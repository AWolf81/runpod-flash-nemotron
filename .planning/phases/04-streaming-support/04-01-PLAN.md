<task type="auto">
  <name>Task 1: Implement SSE streaming pass-through in chat_completions</name>
  <files>nemotron.py</files>
  <action>
Update the `chat_completions` handler in `nemotron.py` to support SSE streaming when `stream=True`.

**Key constraint**: The Flash LB generates a FastAPI app via `create_lb_handler` which calls `app.post(path)(handler)`. FastAPI natively supports `StreamingResponse` returns — the framework passes them through unchanged. The current handler forces `stream=False` with comment "SSE streaming not supported through Flash LB" — this was an unverified assumption.

**Implementation:**

1. Change the return type annotation from `-> dict:` to `-> dict | StreamingResponse:` (import from `starlette.responses` or `fastapi.responses` — both work since fastapi depends on starlette).

2. Add streaming branch: when `stream=True`, use `httpx` streaming to proxy SSE from llama-server:

```python
from starlette.responses import StreamingResponse

if stream:
    async def sse_generator():
        async with httpx.AsyncClient(timeout=1800) as client:
            async with client.stream(
                "POST",
                f"{server_url}/v1/chat/completions",
                json=payload,
            ) as r:
                if not r.is_success:
                    # Yield an error SSE event
                    error_data = await r.aread()
                    yield f"data: {error_data.decode()}\n\n"
                    return
                async for chunk in r.aiter_bytes():
                    if chunk:
                        yield chunk

    # Set stream=True in payload (not forced False)
    payload["stream"] = True
    return StreamingResponse(
        sse_generator(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",  # Disable nginx buffering if any proxy sits in front
        },
    )
```

3. For the non-streaming path, keep the existing `"stream": False` behavior unchanged.

4. Remove the comment `# SSE streaming not supported through Flash LB` and replace with `# stream=True: proxied via StreamingResponse (FastAPI LB supports SSE pass-through)`.

**What to avoid:**
- Do NOT use `r.iter_lines()` — SSE chunks may span multiple lines or contain `data: [DONE]` as a single chunk; `iter_bytes()` preserves the raw SSE wire format llama-server sends.
- Do NOT buffer the full response before yielding — that defeats the purpose.
- Do NOT add `Connection: keep-alive` header manually — FastAPI/uvicorn handles this.

**Verify the fix locally** (the server_ready check path ensures llama-server is running; the streaming branch assumes it's already up by the time the request comes in, since warmup.sh handles cold start). If llama-server returns 503 during a streaming request, the generator should yield an error chunk and close cleanly.

  <name>Task 2: Document streaming support</name>
  <files>docs/streaming.md, README.md</files>
  <action>
Create `docs/streaming.md` documenting streaming support, and update `README.md` with a brief streaming note.

**docs/streaming.md** should cover:

1. **Status**: SSE streaming confirmed working through Flash LB (FastAPI StreamingResponse passes through unchanged).

2. **How it works**: The Flash LB is a FastAPI app. When `stream=True`, `chat_completions` proxies llama-server's SSE stream directly to the client via `StreamingResponse`. No buffering occurs in the Flash worker.

3. **Client configuration**:
   - **curl**: `curl -N -X POST {ENDPOINT_URL}/v1/chat/completions -H "Content-Type: application/json" -d '{"model":"nemotron","messages":[{"role":"user","content":"Hello"}],"stream":true}'`
     The `-N` flag disables curl's own buffering.
   - **Claude Code**: Add `"stream": true` to the body template if using a custom MCP tool, or rely on Claude Code's default streaming behavior (it sends `stream: true` automatically).
   - **OpenCode**: Works with default config — OpenCode sends `stream: true` by default to OpenAI-compatible endpoints.
   - **Python/httpx**: Use `client.stream("POST", url, json=payload)` with `async for chunk in r.aiter_bytes()`.

4. **Known limitation**: `X-Accel-Buffering: no` header is set to prevent proxy buffering, but if RunPod's load balancer introduces its own buffer layer, tokens may still arrive in batches rather than individually. Test with `curl -N` and observe whether chunks arrive token-by-token.

5. **Fallback**: If streaming causes issues (proxy buffering, timeout), set `stream: false` in client config to use buffered mode. Both modes are supported.

**README.md update**: Add a brief "Streaming" section (2-3 lines) after the integration snippets section noting that `stream: true` is supported and pointing to `docs/streaming.md`.
