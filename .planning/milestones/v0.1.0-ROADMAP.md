## Overview

Three phases take this from nothing to a complete OSS developer tool: first build the working deployment scripts (the core product), then add integration guides for each target coding tool, then write the README that makes the whole thing accessible to a first-time user. Phases 4–6 were added during execution: streaming support, model caching (pivoted to binary caching), and warmup performance benchmarking.

## Milestone Summary

**Key Decisions:**

- GGUF via llama-server over vLLM FP8 — single GPU fits budget; 81.7 tok/s gen confirmed
- Claude Code via LiteLLM gateway — Claude Code expects Anthropic-format traffic
- Binary caching on volume over RunPod cached models — RunPod downloads all files, no selective quant
- FastAPI StreamingResponse for SSE — Flash LB passes through unchanged
- Slot priming on first /health ok — eliminates first-request KV cache init failure
- workers=(0,2) scale-to-zero default — cheapest for single-developer

**Issues Resolved:**

- A100 80GB OOM — hardware switched to RTX Pro 6000 Blackwell 96GB
- `--no-kv-offload` OOM — removed; KV cache must share VRAM with model weights
- `--no-mmap` OOM — reverted; CUDA pre-allocates full 79GB buffer before streaming
- First-request KV cache failure — fixed with slot priming in `/health`
- Cold start 16m30s → 8m45s — binary caching on network volume

**Issues Deferred:**

- VRAM load time (~3–6 min): mmap over network volume is slow; no clean fix without more VRAM
- CACHE-01–03: RunPod selective quant support needed before cached models approach is viable
- STRM-03: Claude Code + Mistral Vibe streaming untested against live endpoint

**Technical Debt:**

- `get_cached_model_path()` is dormant code — remove or activate when RunPod ships selective quant support
- `download_model.py` (Phase 1 plan 02) superseded by `python nemotron.py seed`; should be removed or documented as legacy
